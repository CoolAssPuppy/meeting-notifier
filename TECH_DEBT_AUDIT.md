## Executive summary

- **34 findings**: 0 Critical, 9 High, 17 Medium, 8 Low. The repo is in better shape than the size suggests — most of what was Critical in 2026-04-23 (path traversal, prompt injection, JWT trust, Gemini key in URL) is resolved. Remaining debt is structural and dead-code, not security.
- **Largest debt concentration: `AppSettings.swift`** (578 lines + 241-line `+iCloudSync` extension). Adding any new setting touches 5 places; the Sparkle-and-iCloud surface area is unchecked.
- **Three substantial dead-code blocks**: the entire `StoreKitManager` (186 lines, never instantiated), the `WisprEngine` placeholder + enum case (71 lines + 1 unreachable switch arm), and the `micLevel`/`MicLevelBridge` plumbing (4 layers, zero readers).
- **AuthManager.addGoogleAccount and addMicrosoftAccount are 95% identical** — 65 lines of copy-paste that diverge silently when bug-fixed in only one place.
- **Two duplicated calendar-fetch skeletons** (Google, Microsoft) — auth, retry-on-401, status check, JSON cast, parse — same shape both files. `CalendarManagerSupport` was a partial fix; the HTTP loop itself is still copy-pasted.
- **README is materially wrong**: claims macOS 14 + App Sandbox; actual project is macOS 26 + sandbox disabled. Also omits transcription, AI summarization, and Sparkle — three of the most prominent features.
- **A travel-time bug hidden in plain sight**: `CalendarEvent.hasPhysicalLocation` is an English-only heuristic. Non-English locales never see travel-time alerts at all.
- **PostHog API key (`POSTHOG_API_KEY`) is committed to `project.yml:43`**. PostHog write-keys aren't catastrophic, but committing it couples every fork/dev build to your prod project.
- **Three concurrent Timers + a 2-second CoreAudio poll** drive transcription state. Battery drain is real but small; structural smell is bigger than the cost.
- **Tests cover the calm corners** (TranscriptFormatter, AISummarizer, SubfolderResolver) and miss the high-churn ones (AppSettings, AuthManager, CalendarDataManager, TranscriptionCoordinator hot paths).
- **An apparent waveform feature is non-functional**: the mic-level value piped to the UI is permanently zero because the audio thread updates a global `MicLevelBridge` that nothing reads, while `@Published micLevel` only ever resets to 0.

## Architectural mental model

MeetingNotifier is a single-target macOS menu-bar app (XcodeGen-driven, Swift 6 with targeted strict concurrency, App Sandbox disabled, Hardened Runtime on, deployment target macOS 26.0). Three external SPM dependencies: AppAuth (OAuth), Sparkle (auto-update), PostHog (anonymous telemetry).

The runtime is a constellation of `@MainActor`-isolated singletons reached via `.shared`. `AppDelegate` owns the menu-bar status item, popover, native menu, peek window, and transcription banner; it sets up notification observers in `applicationDidFinishLaunching` and acts as the global bus. `AppSettings.shared` is the persistence and iCloud-sync hub for ~30 user preferences plus account list, with `KeychainManager.shared` holding tokens (per-device) and AI API keys (iCloud-synced). `AuthManager.shared` orchestrates AppAuth-driven OAuth flows for Google and Microsoft; `GoogleOAuthManager`/`MicrosoftOAuthManager` are thin per-provider wrappers, with refresh-token logic centralized in `OAuthRefreshSupport`. `CalendarDataManager.shared` polls Google and Graph APIs every 5 minutes (or on account changes) and publishes a filtered `events` array; `NotificationManager.shared` schedules just-in-time UNUserNotifications within a 5-minute lookahead window; `LocationManager.shared` precomputes travel times via MKLocalSearch + MKDirections. The notetaker subsystem is a separate concern: `TranscriptionCoordinator.shared` orchestrates `AudioCaptureManager` (mic via AVAudioEngine), `SystemAudioCapturer` (other participants via ScreenCaptureKit), `EchoDeduplicator`, `SystemAudioEnergyTracker`, and one of three engines (`SpeechAnalyzerEngine`, `DeepgramEngine`, `WisprEngine` — only the first two are wired), then writes a markdown transcript via `TranscriptFormatter` with an optional `AISummarizer` summary. The UI is SwiftUI with a single in-process theme system (`ThemeStore` + 10 `ThemePalette` definitions) and shared components in `SharedComponents.swift`. Drawer-style modals (`SettingsDrawer`, `TranscriptionDrawer`) overlay the main window via `DrawerState.shared`. Telemetry is an explicit indirection: every `Telemetry.capture(...)` call site is provider-agnostic by design.

The README's mental model — MVVM, sandboxed app, macOS 14+, "Settings > Notes" — does not match the code. The actual posture is a SwiftUI-on-AppKit hybrid menu-bar app, deliberately out of the App Store sandbox so it can poll CoreAudio, capture system audio, and access user-chosen folders via security-scoped bookmarks.

## Top 5 — if you fix nothing else, fix these

1. **F001 — Decompose `AppSettings` boilerplate.** This is the file that touches every PR. A `SyncedDefault<T>` property wrapper that registers itself with a settings registry collapses the init readers, the `didSet` writers, the `settingsKeys` array in `+iCloudSync`, and the `applyiCloudChanges` if-key-contains chain into a single line per setting. The diff sketch:
   ```swift
   @SyncedDefault(key: "muteSounds", default: false) var muteSounds: Bool
   ```
   Where `SyncedDefault` reads at init from iCloud-or-UserDefaults, writes both on set (gated by `settingsSyncEnabled` for non-essential keys), and registers itself in a shared `SettingsRegistry.all` so `applyiCloudChanges` can iterate and re-load. ROI: every future setting becomes a one-liner; the privacy-toggle gate (F008) gets enforced uniformly by construction.

2. **F004 + F003 + F005 — Delete the three dead modules.** `StoreKitManager.swift` (186 LOC), `WisprEngine.swift` + `.wispr` enum case + the `case .wispr:` arm + the unreachable subtitle (~85 LOC), and the entire mic-level path (~30 LOC across three files). Combined this is ~300 lines of code that confuses readers and ships in the binary for no behavioral benefit. Verify each with a `grep -rn` first and a `git log --follow` to confirm intent. If the maintainer has a "this will come back" stance for any of them, mark with `// FUTURE:` and an issue link instead of leaving live but unreachable.

3. **F002 + F009 — Collapse the per-provider duplication.** `AuthManager.addGoogleAccount`/`addMicrosoftAccount` (65-line copy) and the calendar-fetch skeletons in `GoogleCalendarManager`/`MicrosoftCalendarManager` (~30 lines per method, twice) are textbook DRY violations of *the same business knowledge*. Fix shape:
   ```swift
   // AuthManager
   private func addAccount(provider: CalendarProvider,
                           manager: any OAuthManagerType,
                           extractEmail: @escaping (OIDAuthState) async -> String?,
                           completion: @escaping (Result<CalendarAccount, Error>) -> Void)
   // CalendarManagerSupport
   static func fetchAuthorized<T>(account: CalendarAccount,
                                  url: URL,
                                  parse: (Data) throws -> T) async throws -> T
   ```
   ROI: a bug fix in retry/backoff or telemetry stops being a copy-paste exercise.

4. **F017 + F018 — Rewrite the README.** The current README describes a different product. New developers grepping for "App Sandbox" or "macOS 14" will form an incorrect mental model. A 30-minute pass against `project.yml` and `find . -name "*.swift" | head -50` produces an honest README. Don't keep both — pick one source of truth.

## Things that look bad but are actually fine

This section is required, per the audit protocol. None of the below were flagged above, despite looking off in isolation.

- **Three concurrent timers in `TranscriptionCoordinator`** (auto-offer 5s + inactivity 10s + auto-save 30s). They look like three things that should be one thing, but their cadences are deliberately decoupled — auto-save's whole point is to be cheap and frequent, inactivity's whole point is to fire only after a long gap, auto-offer's is to recover from a missed `microphoneDidActivate` notification. Combining them couples unrelated concerns.
- **`Task { @MainActor in ... }` re-entries everywhere** (CalendarDataManager `.sink`, NotificationManager UNUserNotificationDelegate, MeetingDetector poll callback, etc.). Looks like anxious main-thread hopping but is required by Swift 6 strict-concurrency when bridging from non-isolated callbacks (timers, KVO observers, delegate methods). The pattern is right.
- **`nonisolated(unsafe)` on `SpeechAnalyzerEngine.analyzer`/`transcriber`/`tasks`.** Looks like a smell. The class isn't `@MainActor` and conforms to a Sendable protocol; the cross-actor reads are real even though the in-class doc says "mutated from @MainActor only" (the doc is misleading). The annotation is correct; the comment isn't.
- **`MicLevelBridge` as a `nonisolated(unsafe) static var Float`.** Crosses the audio thread / UI boundary without a lock. Float reads/writes are atomic on ARM64; the bridge pattern is correct *if* something read it (it doesn't — see F004). The mechanism isn't wrong, the wiring is.
- **Hardcoded `URL(string: "https://...")!` for Google/Graph endpoints.** Standard Swift idiom for compile-known-good URLs; precondition-failing on a typo at app launch is fine. F026 only flags the consistency, not the safety.
- **Empty `noAction(_:)` selector in `AppDelegate+Menu.swift`.** Looks like dead code (and F032 flags it as such for UX), but it's load-bearing: NSMenu auto-disables items with `nil` action *and* their backing `representedObject`. The selector exists so submenu rows render enabled.
- **Sparkle's EdDSA public key in `project.yml:41`.** Looks like a secret. It's not — it's the *public* key Sparkle uses to verify signed update payloads. Has to be embedded in the binary.
- **App Sandbox disabled.** Looks like a security regression. The app needs CoreAudio polling, `ScreenCaptureKit` system audio, and security-scoped folder bookmarks for user-chosen notes folders. The Sparkle/Developer-ID distribution path doesn't go through the App Store sandbox enforcement, so the entitlements are correct for the chosen distribution model. Hardened Runtime is on. Notarization handles the trust story.
- **`CalendarDataManager` 5-minute auto-refresh.** Could be more aggressive. Google and Graph rate-limit aggressively; 5 minutes is the polite ceiling. Real-time would require webhooks, which the menu-bar app posture doesn't support.
- **`@unchecked Sendable` on `EchoDeduplicator`, `SystemAudioStreamOutput`, `ResultState`, `AudioPipelineBox`, `AudioLevelState`.** Looks scary. Each is internally-locked (NSLock or single-thread-write). The Sendable conformance is honest and the locks are real.
